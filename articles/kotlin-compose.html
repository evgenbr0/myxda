Kotlin и Jetpack Compose: современный Android

8 мин чтенияКатегория: AndroidРедакция MYXDA

Современная Android-разработка строится на двух китах: языке Kotlin и декларативном UI-фреймворке Jetpack Compose. Разбираемся, почему индустрия ушла от Java и XML, и как теперь устроены экраны.

Если вы открываете вакансию Android-разработчика в 2026 году, почти наверняка увидите два требования: знание Kotlin и опыт работы с Jetpack Compose. Это не мода, а официальная позиция Google: с 2019 года Kotlin объявлен предпочтительным языком для Android, а Compose стал рекомендуемым инструментом для построения интерфейса. Java и разметка через XML никуда не исчезли — миллионы строк рабочего кода всё ещё на них, — но новые проекты пишут уже иначе.

В этой статье разберём, в чём практическая разница: почему Kotlin удобнее Java, что значит «декларативный UI», как работают composable-функции и состояние, и почему Compose сокращает количество кода и багов. По ходу — небольшой, но полностью рабочий пример.

Почему Kotlin вытеснил Java

Kotlin — язык от компании JetBrains, полностью совместимый с виртуальной машиной Java (JVM). Это значит, что Kotlin и Java спокойно живут в одном проекте: класс на Kotlin вызывает класс на Java и наоборот, можно мигрировать постепенно, файл за файлом. Именно совместимость сделала переход безболезненным. Но удержали разработчиков уже сами возможности языка.

Null-safety: меньше падений

Самая частая ошибка в Java-приложениях — NullPointerException, обращение к объекту, которого нет. Kotlin встраивает защиту от неё прямо в систему типов: тип String не может быть null, а если значение допустимо отсутствующее — это другой тип, String?. Компилятор не даст вам случайно разыменовать nullable-значение без проверки.

val name: String = "Аня"      // null присвоить нельзя
val nick: String? = null      // а здесь можно

// безопасный вызов: вернёт null вместо падения
val len: Int? = nick?.length

// значение по умолчанию через элвис-оператор
val safeLen: Int = nick?.length ?: 0

Краткость

Kotlin убирает шаблонный код. Класс-контейнер данных, который в Java занимает десятки строк с геттерами, сеттерами, equals и hashCode, в Kotlin умещается в одну строку:

data class User(val id: Int, val name: String)

Компилятор сам сгенерирует equals(), hashCode(), toString() и удобный copy(). Плюс выводимые типы (val x = 42 вместо int x = 42), лямбды, функции-расширения и строковые шаблоны ("Привет, $name") делают код заметно компактнее и читаемее.

Корутины: понятная асинхронность

Сетевые запросы, работа с базой и файлами нельзя выполнять в главном потоке — иначе интерфейс «зависнет». Раньше для фоновой работы городили колбэки и вложенные обработчики. Kotlin предлагает корутины — лёгкие «потоки», которые позволяют писать асинхронный код почти как обычный, последовательный:

suspend fun loadProfile(id: Int): User {
    val user = api.fetchUser(id)   // приостановка без блокировки потока
    val avatar = api.fetchAvatar(user)
    return user.copy(avatar = avatar)
}

Ключевое слово suspend помечает функцию, которая может «приостанавливаться» и возобновляться позже, не блокируя поток. Код читается сверху вниз, без лестницы из колбэков.

Совет. Если вы переходите со старого проекта, не переписывайте всё разом. Благодаря совместимости с Java можно начать с новых экранов на Kotlin и Compose, а старые трогать только при необходимости.

Что такое Jetpack Compose

Jetpack Compose — это современный набор инструментов для построения интерфейса Android полностью на Kotlin, без файлов разметки XML. Главное отличие от старого подхода — декларативность.

В классическом (императивном) подходе вы описывали разметку в XML, а в коде вручную находили элементы и меняли их по мере событий: «возьми этот TextView и поставь ему новый текст», «спрячь эту кнопку». Чем сложнее экран, тем легче забыть обновить какой-то элемент — отсюда рассинхронизация интерфейса и состояния.

В декларативном подходе вы описываете, как экран должен выглядеть для текущего состояния, а не последовательность изменений. Когда состояние меняется, Compose сам перерисовывает то, что зависит от этих данных. Формула простая: UI = f(состояние) — интерфейс есть функция от данных.

@Composable
fun Greeting(name: String) {
    Text(text = "Привет, $name!")
}

Аннотация @Composable превращает обычную Kotlin-функцию в «строительный блок» интерфейса. Такие функции можно вкладывать друг в друга, собирая экран как из кубиков, и переиспользовать.

Composable-функции и состояние

Раз UI — это функция от состояния, ключевой вопрос: где хранить данные и как заставить интерфейс реагировать на их изменение. В Compose за это отвечают три понятия: mutableStateOf, remember и рекомпозиция.

  • mutableStateOf создаёт наблюдаемое состояние. Compose следит за любым composable, который читает это значение, и помечает его на перерисовку при изменении.
  • remember сохраняет значение между перерисовками. Без него состояние создавалось бы заново при каждой рекомпозиции, и счётчик всегда сбрасывался бы в начало.
  • Рекомпозиция — это повторный вызов composable-функций при изменении данных, которые они читают. Compose умён: он перевызывает не весь экран, а только те функции, чьи входные данные действительно поменялись.

На практике их объединяют так:

var count by remember { mutableStateOf(0) }

Здесь count ведёт себя как обычная переменная: читаем count, пишем count = count + 1. Но за кулисами это наблюдаемое состояние, переживающее рекомпозиции, и любое изменение автоматически обновит зависящий от него интерфейс.

Совет. Если состояние должно пережить поворот экрана, используйте rememberSaveable вместо remember — оно сохранит значение при пересоздании активности.

Пример: счётчик на Compose

Соберём всё вместе. Ниже — полноценный экран со счётчиком: текст показывает текущее значение, кнопка увеличивает его на единицу. Обратите внимание, что нигде нет команды «обнови текст» — мы лишь меняем состояние, а перерисовку Compose берёт на себя.

import androidx.compose.foundation.layout.*
import androidx.compose.material3.Button
import androidx.compose.material3.Text
import androidx.compose.runtime.*
import androidx.compose.ui.Alignment
import androidx.compose.ui.Modifier
import androidx.compose.ui.unit.dp

@Composable
fun CounterScreen() {
    // Наблюдаемое состояние, переживающее рекомпозицию
    var count by remember { mutableStateOf(0) }

    Column(
        modifier = Modifier.fillMaxSize().padding(24.dp),
        horizontalAlignment = Alignment.CenterHorizontally,
        verticalArrangement = Arrangement.Center
    ) {
        Text(text = "Нажатий: $count")

        Spacer(modifier = Modifier.height(16.dp))

        Button(onClick = { count++ }) {
            Text(text = "Нажми меня")
        }
    }
}

Что здесь происходит по шагам: при первом запуске count равен нулю и на экране видно «Нажатий: 0». По нажатию на кнопку срабатывает onClick, который увеличивает count. Поскольку Text читает count, Compose помечает именно его на рекомпозицию и перерисовывает с новым значением. Column, Button и прочее, что от count не зависит, не трогается.

Чем Compose лучше старого подхода

Главный выигрыш — единый источник правды. Состояние живёт в одном месте, а интерфейс автоматически ему соответствует, поэтому исчезает целый класс ошибок «забыли обновить элемент». Заодно убирается шов между XML и кодом: больше не нужно искать вьюхи по идентификаторам и связывать их вручную.

АспектView / XMLJetpack Compose
ПодходИмперативный: меняем элементы вручнуюДекларативный: описываем результат
РазметкаОтдельные XML-файлыВсё на Kotlin, в одном месте
Поиск элементовПо id (findViewById / binding)Не требуется
Обновление UIВручную при каждом событииАвтоматически по состоянию
ПереиспользованиеСложнее (кастомные View, include)Обычные функции

Кода в итоге меньше, он сосредоточен в одном файле и читается линейно. Это не значит, что Compose решает все задачи лучше: у зрелого XML-подхода огромная экосистема и масса готовых решений, а Compose требует более внимательной работы с производительностью рекомпозиции. Но для новых проектов вектор индустрии однозначен.

Экосистема: Material 3, навигация, превью

Compose — это не только базовые «кубики», но и развитая обвязка вокруг.

Material 3

Библиотека material3 приносит готовые компоненты в стиле актуального дизайна Google: кнопки, карточки, поля ввода, верхние панели, нижнюю навигацию. Через MaterialTheme задаются цветовая схема, типографика и формы, а поддержка динамических цветов подстраивает палитру под обои пользователя.

Навигация

Библиотека Navigation Compose позволяет описывать переходы между экранами декларативно — через граф маршрутов и NavController, который управляет стеком экранов и передачей аргументов между ними.

@Composable
fun AppNav(navController: NavHostController) {
    NavHost(navController, startDestination = "home") {
        composable("home") { HomeScreen(navController) }
        composable("counter") { CounterScreen() }
    }
}

Превью прямо в IDE

Одна из самых любимых разработчиками возможностей — аннотация @Preview. Она отрисовывает composable в Android Studio без запуска приложения на устройстве или эмуляторе: меняете код — видите результат в панели рядом.

@Preview(showBackground = true)
@Composable
fun CounterScreenPreview() {
    CounterScreen()
}

Можно сделать несколько превью с разными данными, темами или размерами экрана и видеть их одновременно — это резко ускоряет проработку интерфейса.

Коротко

  • Kotlin — официально предпочтительный язык Android: null-safety снижает число падений, синтаксис короче, корутины упрощают асинхронность, а совместимость с Java позволяет мигрировать постепенно.
  • Jetpack Compose строит интерфейс декларативно на Kotlin без XML по принципу «UI = функция от состояния».
  • Состояние держится на связке mutableStateOf + remember, а обновляет UI рекомпозиция — повторный вызов только тех функций, чьи данные изменились.
  • По сравнению с View/XML Compose убирает ручной поиск и обновление элементов, сокращает код и устраняет рассинхрон UI и данных.
  • Вокруг Compose выросла экосистема: Material 3, Navigation Compose и мгновенные превью @Preview прямо в Android Studio.
